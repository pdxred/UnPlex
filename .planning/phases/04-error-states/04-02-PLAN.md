  <action>
**PlexApiTask changes:**
- Ensure PlexApiTask.xml has a `responseCode` interface field (type integer, default 0) if not already present
- In PlexApiTask.brs, after the HTTP request completes, set `m.top.responseCode = responseCode` so screens can distinguish network errors (responseCode less than 0) from HTTP errors (4xx/5xx)

**Retry pattern for each screen (HomeScreen, DetailScreen, EpisodeScreen, SearchScreen):**

Replace each screen's existing `showError()` function with a new error handling system:

1. Add `m.retryCount = 0` in `init()`
2. Add `m.retryContext = invalid` in `init()` to store the last request's parameters (endpoint, params, callback handler string)

3. Before each API request, store the retry context:
   ```
   m.retryContext = { endpoint: endpoint, params: params, handler: "onApiTaskStateChange" }
   ```
   (Adapt field names to match each screen's actual task setup pattern)

4. In each task state change callback, when state = "error":
   - If `m.retryCount = 0`: increment `m.retryCount` to 1, call `retryLastRequest()` (silent auto-retry, no dialog)
   - If `m.retryCount >= 1`: reset `m.retryCount = 0`, hide spinner, call `showErrorDialog()` with contextual message

5. Add `retryLastRequest()` sub that creates a NEW task node (never reuse failed task -- per research pitfall 3), sets endpoint/params from `m.retryContext`, observes the handler, and runs it. Shows spinner.

6. Add `showErrorDialog(title, message)` sub:
   - Per user decision: Network errors as dialog overlays (modal)
   - Create StandardMessageDialog with title and message
   - Buttons: ["Retry", "Dismiss"]
   - Observe `buttonSelected` and `wasClosed`
   - Per research anti-pattern: Check `m.top.getScene().dialog <> invalid` before creating -- if a dialog is already open, queue or skip
   - Contextual messages per user decision. Use screen-specific messages:
     - HomeScreen: "Couldn't load your library"
     - DetailScreen: "Couldn't load details"
     - EpisodeScreen: "Couldn't load episodes"
     - SearchScreen: "Search failed"

7. Add `onErrorDialogButton(event)` handler:
   - Button index 0 (Retry): close dialog, call `retryLastRequest()`
   - Button index 1 (Dismiss): close dialog, call `showInlineRetry()`

8. Add `onErrorDialogClosed(event)` handler:
   - Restore focus to the appropriate screen element (grid, list, etc.)
   - Per research pitfall 4: Always explicitly setFocus after dialog closes

**Inline retry (per user decision: after dismissing error dialog, stay on screen with inline Retry option):**

9. Add to each screen's XML a `retryGroup` (visible=false):
   - Label with the error message (e.g., "Something went wrong")
   - ButtonGroup with single button ["Retry"]
   - Position centered in content area

10. Add `showInlineRetry()` sub: hide grid/content, show retryGroup, setFocus on retryButton
11. Observe retryButton's `buttonSelected` -- on press, hide retryGroup, call `retryLastRequest()`

Per user decision: Auth token expiry treated same as any other error -- no special auth flow, just the standard error dialog with retry.

On successful retry (state = "completed"): reset `m.retryCount = 0`, hide retryGroup if visible, process response normally.

BrightScript limitation: Cannot pass functions as callbacks. Store retry context as an associative array with string handler name. Use the screen's existing task setup pattern to re-create the request.

  <name>Task 2: Add server disconnect detection and reconnect flow in MainScene</name>
  <files>

  <action>
**Global server unreachable signal:**

1. In MainScene.brs `init()`, add a global field `m.global.addFields({ serverUnreachable: false })`
2. Observe `m.global.observeField("serverUnreachable", "onServerUnreachable")`

**PlexApiTask integration (already partially done in Task 1):**
The screens' error handlers should set `m.global.serverUnreachable = true` when they detect a network-level error (responseCode less than 0) AND the silent auto-retry also fails. This signals MainScene to take over with the disconnect flow.

3. In each screen's error handler (from Task 1), after the second failure: check if `m.currentApiTask.responseCode < 0` (network error, not HTTP error). If yes, set `m.global.serverUnreachable = true` instead of showing the per-screen error dialog. If responseCode >= 400 (HTTP error), show the per-screen error dialog as normal.

**Server disconnect dialog (in MainScene):**

4. Add `onServerUnreachable(event)` in MainScene.brs:
   - Per user decision: Silent background retry first
   - Create a ServerConnectionTask to test current server connectivity
   - If ServerConnectionTask succeeds (server is back): set `m.global.serverUnreachable = false`, signal current screen to re-fetch (see step 7)
   - If ServerConnectionTask fails: show disconnect dialog

5. Add `showServerDisconnectDialog()`:
   - StandardMessageDialog with title "Server Unreachable"
   - Message: "Can't connect to your Plex server. Check your network connection and try again."
   - Buttons: ["Try Again", "Server List"] (per user decision)
   - Observe buttonSelected and wasClosed

6. Add `onDisconnectDialogButton(event)`:
   - Button 0 ("Try Again"): close dialog, run ServerConnectionTask again to test connectivity
     - If successful: set `m.global.serverUnreachable = false`, signal re-fetch
     - If failed: show disconnect dialog again
   - Button 1 ("Server List"): close dialog, navigate to ServerListScreen (pop all screens, push ServerListScreen)
   - Per research pitfall 4: restore focus after dialog closes

**Resume on reconnection (per user decision: resume user's position, re-fetch data):**

7. Add a global field `m.global.addFields({ serverReconnected: false })`. When server connectivity is restored, set `m.global.serverReconnected = true`. Each content screen should observe this field in their `init()` and re-trigger their data fetch when it fires. After re-fetch triggers, reset it to false.

8. In each screen's init() (HomeScreen, DetailScreen, EpisodeScreen, SearchScreen), add:
   ```
   m.global.observeField("serverReconnected", "onServerReconnected")
   ```
   And add `onServerReconnected` sub that calls the screen's existing data-loading function (e.g., `loadLibrary()`, `loadMetadata()`, `loadSeasons()`).

**Edge cases:**
- Do not show server disconnect dialog if a dialog is already open (check `m.top.dialog <> invalid`)
- Do not show server disconnect dialog during playback (check if VideoPlayer is the active screen -- per research pitfall 5, playback errors are a separate concern)
- Reset `m.global.serverUnreachable = false` before the reconnect attempt to prevent duplicate triggers
